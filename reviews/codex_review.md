**Block A — Portfolio Feasibility**

A1. Это частично велосипед. Generic NL->SQL уже не даёт сильного сигнала: Vanna позиционируется как open-source RAG/SQL agent с multi-db и access control ([vanna.ai](https://vanna.ai/), [GitHub](https://github.com/vanna-ai/vanna)); WrenAI уже закрывает text-to-SQL/text-to-chart через semantic layer и 12+ источников ([GitHub](https://github.com/Canner/WrenAI)); DataHerald давно делает enterprise NL->SQL ([GitHub](https://github.com/Dataherald/dataherald)); LangChain SQLDatabaseToolkit даёт готовый agent loop со schema lookup/query/checker и прямо предупреждает о рисках write/expensive queries ([docs](https://docs.langchain.com/oss/python/integrations/tools/sql_database/)); PandasAI закрывает NL data analysis + charts для табличных данных ([docs](https://docs.pandas-ai.com/v3/chat-and-output)). Поэтому "чат, который пишет SQL и строит график" будет выглядеть как учебная сборка существующих паттернов. Сигнал появится только если проект будет не про наличие NL->SQL, а про инженерно честную оценку качества, schema linking, безопасное исполнение и разбор ошибок на реальных БД.

A2. Отличимыми могут быть только 2-3 вещи. Первое: публичный eval с ablation table, где видно вклад schema retrieval, few-shot, repair loop и safety gates. Второе: real-world analyst workflow на StackExchange mini с хорошими gold questions, где SQL, результат, объяснение, cost/latency и failure mode показаны прозрачно. Третье: safety/execution layer, который демонстрирует DE-зрелость: read-only role, AST validation, timeouts, result caps, cost checks и traceability. Generic можно резать: chat UI, history/bookmarks, DB switcher как feature, Prometheus/OTel, Redis, live `/eval`, Chinook как showcase, auto chart selection в полном объёме. Это не отличает автора от Vanna/WrenAI/LangChain.

A3. Mistral-only допустим как осознанный эксперимент, но плохо как жёсткая архитектурная зависимость. В 2026 это легко считывается как vendor-lock demo, особенно если `codestral-2501` захардкожен: в актуальных Mistral docs он уже listed как legacy/deprecated-era API name, а линейка обновилась ([Mistral models](https://docs.mistral.ai/models), [Codestral docs](https://docs.mistral.ai/capabilities/code_generation/)). Ответ на "почему не GPT-4 / Claude / local" должен быть не идеологическим, а измерительным: "Mistral chosen as default because Codestral is code-specialized, cheaper/faster for SQL generation, and avoids another OpenAI clone; provider interface is isolated; here is a 30-question bakeoff against one frontier and one local/open model." Без такого bakeoff это выглядит как вкусовщина, не как senior engineering choice.

A4. BIRD-mini + StackExchange Postgres уместны, Chinook почти не нужен. BIRD даёт публичную метрику и сложные схемы; официальный сайт описывает full BIRD как 12,751 Q-SQL pairs, 95 DB, 37 domains, 33.4 GB ([BIRD](https://bird-bench.github.io/)). Но в архитектуре есть несостыковка: BIRD Mini-Dev на официальной странице описан как 500 examples, а не 1500/11 DB. Это надо исправить до README, иначе eval section будет выглядеть неряшливо. StackExchange хорош как "аналитическая" БД, но full dump может съесть время на ingestion/indexing вместо NL->SQL. Более высокий impact при той же сложности: curated StackExchange mini с 20-30 gold analyst questions, или DuckDB/Postgres analytics dataset типа NYC Taxi / TPC-H / ClickBench-scale subset, где можно показать performance constraints и реальные aggregation/time-series паттерны. Chinook оставить только для smoke/local demo, не продавать как портфолио-сигнал.

A5. Execution Accuracy >=50% на BIRD-mini с `codestral-2501` zero-shot выглядит завышенно. В prompt смешаны разные вещи: Codestral 25.01 blog даёт Spider 66.5, но это не BIRD ([Mistral blog](https://mistral.ai/news/codestral-2501)); официальный BIRD Mini-Dev leaderboard показывает GPT-4 около 47.8/40.8/35.8 EX на SQLite/MySQL/PostgreSQL, а TA+GPT-4 около 58.0/49.2/50.8 ([BIRD Mini-Dev](https://bird-bench.github.io/)). Значит 50% честно возможно только на удачном dialect/subset и не zero-shot: нужен schema linking, few-shot, maybe candidate reranking, repair and careful evaluation. Для solo builder реалистичнее поставить baseline target 35-45% first pass, stretch target 50% after ablations, and report by dialect/difficulty.

**Block B — Architecture Review**

B1. LangGraph на 11 узлов раздут. `classify_intent` часто дублирует то, что генератор SQL и renderer всё равно должны вывести из вопроса. `select_database` лишний, если UI уже даёт explicit DB; auto-select полезен только как optional convenience. `retrieve_schema` и `retrieve_examples` лучше объединить в context builder с единым budget и trace. `choose_format` должен быть deterministic от result shape, не отдельный LLM-heavy decision. `persist_trace` не обязан быть graph node; это middleware/span. Не хватает: clarification path для неоднозначных вопросов, dialect adapter, schema-linking confidence, candidate generation/ranking, deterministic result-shape contract, refusal path for unsafe/expensive queries, and error taxonomy. Retry на `static_validate` и `execute` нужен, но один repair loop достаточно. Retry на `verify_result` опасен: empty result может быть правильным, а "аномально много null" не доказывает ошибку SQL. EXPLAIN failure должен чаще возвращать диагностический refusal/ask-to-narrow, а не просить LLM "попробовать ещё".

B2. Schema-RAG из 4 Chroma collections частично оправдан, но relations в vector DB выглядят неправильно. FK/relations лучше хранить как deterministic catalog graph и расширять retrieved tables через graph traversal. `schema_columns` отдельной коллекцией может помочь на wide tables, но сначала надо доказать, что она лучше compact table cards. Correct baseline: full schema prompt on small DB; table-level BM25/dense retrieval; table cards with columns/samples/stats; table cards + FK expansion; then few-shot retrieval. Только после этого можно говорить, что 4 коллекции нужны. Few-shot Q->SQL retrieval имеет смысл, но надо запретить leakage из eval dev examples и показывать exact train/dev split hygiene.

B3. 4-layer SQL safety не hype, но EXPLAIN-gate в текущем виде слабый и brittle. Mandatory layers: DB role with SELECT-only/read-only transaction, AST allowlist, statement timeout/result cap. EXPLAIN-gate полезен на StackExchange, но `Total Cost > X` плохо переносится между DBs, зависит от stats and planner settings, и будет фальшиво банить нормальные аналитические запросы. Оставшиеся attack vectors: expensive SELECT через cross joins/window functions/generate_series/recursive CTE; `pg_sleep` and other functions if not blocked; metadata exfiltration via `pg_catalog`/`information_schema`; SQLite-specific `ATTACH`, extension/load/readfile vectors; prompt injection through sample values/schema comments; huge result payloads into UI/chart renderer; temp file exhaustion; lock/IO pressure from long scans. Для Postgres нужно явно фиксировать `search_path`, `default_transaction_read_only`, `statement_timeout`, `idle_in_transaction_session_timeout`, `temp_file_limit`, function allowlist, schema allowlist and row/result caps.

B4. Execution Accuracy + Component Match недостаточно. Нужны at least: valid SQL rate, execution failure rate, timeout rate, empty-result rate, schema-linking recall, exact result equivalence with order-insensitive comparison where allowed, partial-credit/Soft-F1 for near misses, cost per query, token usage, p50/p95 latency, retry rate, repair success rate, EXPLAIN rejection rate, and R-VES/efficiency for expensive SQL. Для portfolio особенно важны error slices: by difficulty, by dialect, by join count, by aggregation/window/time filter. Smoke-eval на 50 examples in CI реалистичен только с cached LLM responses or frozen golden outputs. Live Mistral API on every merge will be flaky, slow, and rate-limit/cost sensitive. CI should run static unit tests + 5-10 cached smoke examples; nightly/manual can run 50; full eval stays manual with a published report.

B5. Multi-format render с Vega-Lite spec от LLM будет работать на красивых demos, но как core feature ненадёжен. Vega-Lite is a JSON spec language with required `data`, `mark`, `encoding`, schema/version constraints and validation hooks ([Vega-Lite spec](https://vega.github.io/vega-lite/docs/spec.html)); LLMs will still mismatch field names, infer wrong temporal type, aggregate twice, sort incorrectly, choose pie for high-cardinality categories, emit invalid JSON, or generate a misleading chart for already-aggregated data. Better design: deterministic renderer owns the spec; LLM may choose `chart_intent` from a small enum and write a caption. Validate spec against Vega schema, whitelist marks/channels/transforms, cap categories, force explicit x/y fields from result metadata, and fall back to table if confidence is low.

B6. С делением в section 12 согласен только наполовину. "Нагруженное, но не фейковое": eval harness, schema retrieval, few-shot retrieval if split-safe, SQL guards, result tracing/show-working, and a minimal UI. "Похоже на production-overhead for demo": Redis, Prometheus + OpenTelemetry + Langfuse all together, backend history/bookmarks, dynamic eval page, multi-DB auto-switching, full multi-format rendering, testcontainers everywhere, and a 10+ node graph. У автора уже есть RAG_SA с FastAPI/LangGraph/Chroma/Postgres/Langfuse/RAGAS/mypy strict. Повтор того же стека здесь не добавит новый сигнал; нужен measured NL->SQL competence, not another infra stack.

B7. Roadmap 9 stages / 5-6 weeks is optimistic. The stage that will blow the timeline is not UI; it is eval harness + dataset setup + error analysis. Getting BIRD DBs, dialects, timeouts, result equivalence, order normalization, numeric tolerance, and reproducible reports right is tedious. StackExchange ingestion/indexing can also consume a week by itself if full dump is attempted. Realistic timeline for a polished portfolio piece is 8-10 focused weeks, or 5-6 weeks only after cuts: no Redis/Prom/OTel, no live eval page, no full dump, no LLM-authored Vega, no backend history.

B8. Risk section misses more likely risks. Biggest: benchmark leakage from using dev examples as few-shot; evaluation bugs that inflate EX; model/API churn and deprecation of pinned model names; BIRD dialect mismatch and inconsistent DB setup; prompt injection through database contents; LLM repair making wrong-but-executable SQL more confidently wrong; empty-result retry corrupting correct negatives; local resource limits from Chroma + Postgres + StackExchange; latency/cost spikes from multi-candidate generation; and demo credibility risk if charts are pretty while SQL correctness is weak. Also missing: "business semantics" risk. NL->SQL often fails because "active user", "growth", "top tag" and "revenue" are definitions, not schema names. WrenAI's semantic-layer positioning exists for exactly this reason.

**Block C — Concrete Fixes**

1. Add an ablation matrix before locking architecture because without `full_schema`, `BM25/table cards`, `Chroma table+column`, `+fewshot`, `+repair` comparisons the central Schema-RAG claim is unproven.

2. Replace hard Mistral-only coupling with a provider interface plus a 30-question bakeoff because this turns the vendor-lock objection into a measured tradeoff instead of a preference.

3. Replace `BIRD-mini + full StackExchange + Chinook` with `BIRD Mini-Dev + curated StackExchange mini + optional Chinook smoke` because full StackExchange and toy Chinook both distract from the measurable portfolio signal.

4. Replace the 11-node graph and three retry points with a lean pipeline (`context_builder -> generate_sql -> validate/repair_once -> execute -> deterministic_format -> explain/trace`) because scattered retries increase complexity and can hide wrong-but-executable SQL.

5. Replace LLM-generated Vega-Lite specs with deterministic chart templates plus LLM-selected chart intent/caption because invalid or misleading charts will otherwise become a high-probability demo failure unrelated to the core NL->SQL skill.

**Block D — Final Verdict**

Worth building for a Senior DE portfolio only if it is reframed as a measured NL->SQL evaluation/safety project with lean architecture, honest baselines, curated real DB questions, provider isolation, and deterministic rendering; otherwise it is another generic chat-with-database demo.
